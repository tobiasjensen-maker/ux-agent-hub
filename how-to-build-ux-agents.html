Using Jobs to Be Done
to Design Agents

A framework for deciding what agents to build — and a case study from our UX team.

Tobias Jensen

The theory

What is Jobs to Be Done?

JTBD is a framework that shifts focus from what you're building to why someone needs it. Instead of starting from features or solutions, you start from the jobs people are trying to get done.

The idea

People don't want products — they want to make progress in their lives or work. A "job" is the progress someone is trying to make in a particular circumstance. When you understand the job, you can design something that actually helps.

The job statement

Every job can be expressed as a single statement that captures context, motivation, and desired outcome. This gives you a stable anchor that doesn't change even as tools and solutions evolve.

The job statement format "When [situation], I want to [motivation], so I can [expected outcome]."

The method

Jobs to Be Done tells you what agents to build

The same framework works for agent design. An agent is most useful when it helps someone complete a job they already have. Define the job, define the agent.

1

Job = Agent

Each job becomes one agent. The job statement becomes the agent's purpose.

→
2

Steps = Instructions

The numbered steps within each job become the agent's numbered instructions.

→
3

Templates = Output

The templates you already use become the agent's output format.

The core insight Don't ask "what can AI do?" — ask "what jobs do people in this role need to get done?" Then build agents for those jobs.

Case study:
UX Design

We applied this method to UX design at e-conomic. Here's what we found — and what we built.

Case study · The problem

UX work is full of manual artifact creation

Between the creative moments, designers spend most of their time producing structured documents — research plans, pitches, problem definitions, briefs, prototypes, test scripts, handoff specs. Each one follows a pattern.

Artifact after artifact

Research briefs, opportunity write-ups, pitch decks, design briefs, prototype scaffolds, validation plans, delivery specs — each project produces dozens of structured documents that all follow known templates.

Most of it is assembly

The real thinking is a fraction of the effort. The rest is gathering context from previous artifacts, filling in templates, and making sure nothing gets lost along the way.

This is where agents help

If an artifact follows a template and draws on existing context, an agent can draft it. JTBD gave us the method to decide which ones to build.

Case study · The opportunity

The 10 / 80 / 10 rule

There are two moments in any design project where AI has no place: when you define the problem, and when you decide if the work is ready to ship. Everything in between? That's where agents earn their keep.

10%You
80%Agents
10%You
Why this matters When you let AI handle the structured middle, you protect the parts that make you a designer — while reclaiming hours spent on context-gathering and template-filling. The agents don't replace your creativity. They clear the path for it.

The Job Map

Discovery

Research a Question ▾

"When I have a question about how our users work, I want to find the answer so I can make evidence-based design decisions."

  1. Search existing research for prior work
  2. Assess what's known and identify gaps
  3. Refine the research question
  4. Choose a research method
  5. Create a discovery item in the backlog
  6. Write a research brief with interview questions

Recruit Participants ▾

"When I have a research brief, I want to quickly shortlist relevant users."

  1. Parse participant criteria from the brief
  2. Search user panel by segment
  3. Search Airtable support tickets
  4. Search sales/onboarding data
  5. Score and rank candidates
  6. Build shortlist — primary + contrast segments
  7. Draft recruitment email

Find Opportunities ▾

"When I've completed research, I want to extract validated opportunities."

  1. Analyse research input
  2. Map to JTBD framework
  3. Assess mission alignment
  4. Extract core opportunity
  5. Draft opportunity title & description
  6. Link to discovery item
  7. Add to backlog
Delivery & Design

Write a Pitch ▾

"When I have a validated opportunity, I want to articulate the business case."

  1. Read the opportunity
  2. Trace upstream to JTBD
  3. Fill Pitch Template v3.0
  4. Present for review
  5. Save to Pitches collection

Create a Design Brief ▾

"When a pitch is approved, I want to create a thorough brief."

  1. Trace pitch → opportunity → JTBD → mission
  2. Fill Design Brief 2.1 template
  3. Flag gaps with [NEEDS INPUT]
  4. Structure for LLM context
  5. Present for review
  6. Save to Design Briefs collection

Generate Prototypes ▾

"When I have a brief, I want to explore multiple solution directions as prototypes."

  1. Read the design brief
  2. Identify the strategic proposition
  3. Scaffold Vite + React + Taco project
  4. Generate 2–3 alternatives
  5. Build to standalone HTML
  6. Document design rationale
Validation & Iteration

Validate with Users ▾

"When I have prototypes, I want to structure how I'll test them."

  1. Analyse the solution(s)
  2. Recommend validation method
  3. Generate test script
  4. Define participant criteria
  5. Create feedback template
  6. Set success criteria

Iterate on Feedback ▾

"When I have validation feedback, I want to feed it back into the design."

  1. Structure the feedback
  2. Update the design brief
  3. Regenerate solutions
  4. Compare new vs. previous

Package for Delivery ▾

"When design is validated, I want to package a handoff-ready spec."

  1. Read the validated brief
  2. Generate interaction specs
  3. Map to Taco components
  4. Document tokens
  5. Attach validation evidence
  6. Define acceptance criteria

From jobs
to agents

The steps you already follow become the agent's instructions. Here's how we translated one UX job into a working agent.

Case study · Translation

"Research a Question" → Discovery Agent

Left: what the designer does. Right: what the agent does.

Designer's steps

  1. Search existing research for prior answers
  2. Assess what's known and what gaps remain
  3. Refine the research question
  4. Choose a research method
  5. Create a discovery item in the backlog
  6. Write a research brief with interview questions

Agent's instructions

  1. Search All Discovery and Opportunities collections
  2. If found, present findings and ask if new research is needed
  3. Refine input into a specific research question
  4. Suggest the best method (interview, data, diary, usability, survey)
  5. Add item to All Discovery with JTBD, method, status, owner
  6. Append research brief, 5–8 questions, participant criteria
Notice The instructions are almost identical to the steps. The agent just needs the addresses — collection IDs, column IDs, field names.

Case study · Wiring

Inputs, tools, and handoffs

Inputs

What does the designer need before starting? A JTBD selection and a research question. These become the agent's input form — the fields to fill before hitting "Deploy".

Tools

What systems does the designer touch? Slite, Airtable, user panel, Figma. These become the agent's integrations — APIs and MCP connections.

Handoffs

What does the next step need? The Discovery Agent's output feeds the Recruitment Agent's input. This chaining turns individual agents into a pipeline.

Apply it yourself

The method works for any role

UX was our case study — but the recipe is the same for sales, support, engineering, ops, or any team with structured, repeating work.

1. Pick a role

Who does structured, repeating work? That's your starting point.

2. Map the jobs

What are the 5–10 jobs someone in that role needs to get done?

3. Write down the steps

What do they open, read, fill in, save? Be specific.

4. List the tools

What systems do they touch? These become the agent's integrations.

5. Build the agents

Steps become instructions. Templates become output format. The manual process IS the prompt.
